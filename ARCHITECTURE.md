## 1. Текущая обстановка (март 2026)

### Что блокирует ТСПУ
- **WireGuard** — детектируется за секунды, блокировка 100%
- **OpenVPN** — детектируется, блокировка 100%
- **Shadowsocks / Outline** — блокировка ~95%
- **VLESS с популярными SNI** (Google, Yandex, VK, Bing) — блокируется
- **VLESS Reality с малоизвестными SNI** — работает, но ТСПУ учится
- **XHTTP (новый транспорт)** — работает, следующий шаг после TCP
- **AmneziaWG 2.0** — работает через рандомизацию заголовков

### Белые списки мобильного интернета
С осени 2025 мобильные операторы (МТС, Билайн, Мегафон, Теле2) при «шатдаунах»
оставляют доступ только к одобренным доменам. Блокировка двухуровневая:
- **По SNI** — проверяется домен, к которому идёт подключение
- **По IP/CIDR** — проверяется, принадлежит ли IP сервера белому списку

Операторы переходят на комбинированную модель IP + SNI.

### Белый список доменов (SNI) для мобильных операторов
Источник: github.com/hxehex/russia-mobile-internet-whitelist

Российские домены:

## 2. Архитектура: два уровня

### Уровень 1 — Прямое подключение (80% юзеров)
Для: проводной интернет, мобильный без активных белых списков

```
Юзер (Happ) → VLESS Reality → Зарубежный сервер → Интернет
```

Протокол: VLESS + XTLS-RPRX-VISION + Reality
Транспорт: TCP (основной) + XHTTP (fallback)
SNI: малоизвестные зарубежные домены (НЕ Google/Yandex/VK)
Fingerprint: Chrome (uTLS)
Порт: 443

#### Правила выбора SNI для уровня 1
НЕЛЬЗЯ использовать: google.com, yandex.ru, vk.com, bing.com, microsoft.com,
apple.com, icloud.com — все палятся ТСПУ.

ТСПУ проверяет: SNI = X, но IP сервера ≠ IP реального сервера X → блокировка.
На крупных хостингах (Hetzner, DO) можно подобрать SNI сайтов,
которые тоже хостятся у этого провайдера, чтобы ASN совпадал.

Рабочие подходы:
1. Использовать RealiTLScanner на VPS для поиска SNI в той же подсети
2. Малоизвестные зарубежные сайты: pdf24.org, deepl.com, booking.com
3. Свой домен + Nginx + Certbot (самый надёжный для долгосрочной работы)
4. CDN-домены (ampproject.org, jsdelivr.net) — но могут палить

Кандидаты для SNI (проверять для конкретного хостера):
- pdf24.org (подтверждён на Hetzner в комментариях Хабра)
- deepl.com (использует TunnelGuard)
- booking.com (использует TunnelGuard)
- telegram.org, whatsapp.com (рекомендуют на форумах)
- Свой .com домен с Nginx + Let's Encrypt (лучший вариант)

### Уровень 2 — Через российский relay (20% юзеров)
Для: мобильный интернет при активных белых списках (шатдауны)

```
Юзер (Happ) → Российский relay (белый IP) → Зарубежный exit → Интернет
```

#### Российский relay-сервер
Задача: принимать трафик от юзера (ТСПУ пропускает, т.к. IP в белом списке)
и пробрасывать на зарубежный exit-сервер.

Где арендовать:
- VK Cloud (подсеть 212.233.x.x) — IP в белом списке у Мегафона
- Яндекс.Облако (подсеть 51.250.x.x) — IP в белом списке
- Reg.ru, Selectel — российские хостеры, менее контролируемые чем VK/Ya

РИСК: российский хостер может заблокировать аккаунт за VPN-трафик.

#### Маскировка российского relay (ваша идея!)
На relay-сервере ставим НАСТОЯЩИЙ работающий сайт:
1. Сайт проверки IP-адреса (speedtest/ping/IP lookup) — объясняет трафик
   с разных IP, потому что люди заходят проверять свой IP
2. XRay слушает на том же порте 443
3. Если запрос без правильного конфига → показываем сайт
4. Если с конфигом → пробрасываем VPN-трафик

Reality masquerade + реальный легитимный сайт = двойная защита.

Хостер видит: обычный веб-сервер с посетителями и HTTPS-трафиком.
РКН видит: соединение к российскому IP с SNI из белого списка.

#### Зарубежный exit-сервер
Задача: выход в свободный интернет.
Relay подключается к exit через VLESS Reality.
Exit стоит в Европе (Hetzner, DigitalOcean).

## 4. Балансировка нагрузки (Smart Load Balancer)

### Принцип: все серверы работают всегда
Нет «основных» и «резервных». Все 4 зарубежных сервера активны и обслуживают
юзеров одновременно. Нагрузка размазана равномерно.

### Двухуровневая балансировка

#### Уровень A — Клиентский (Happ автопинг)
Subscription содержит ВСЕ серверы. Happ сам пингует и выбирает лучший.
Юзер ничего не делает — приложение решает за него.

#### Уровень B — Серверный (API Smart Balancer)
API каждые 30 секунд проверяет все серверы:
- Ping (доступность из РФ)
- Нагрузка CPU/RAM
- Количество активных подключений
- Оставшийся трафик в месяце
- Скорость (iperf тест)

На основе этих данных API формирует subscription:
- Серверы отсортированы по «здоровью» (лучший первый)
- Перегруженные опускаются вниз
- Недоступные удаляются из списка
- При неравномерной нагрузке — новые юзеры направляются на менее нагруженный

#### Формула распределения
Каждый сервер получает «вес» от 0 до 100:
  вес = (100 - CPU%) × 0.4 + (100 - RAM%) × 0.2 + (ping_ok ? 30 : 0) + (трафик_остаток > 20% ? 10 : 0)

Юзеры распределяются пропорционально весу.
Пример: HEL=85, FAL=70, AMS=60, FRA=50 → HEL получит ~32%, FAL ~26%, AMS ~23%, FRA ~19%

### Сценарии

#### Нормальный режим
- 100 юзеров онлайн
- HEL: ~25 юзеров (ближе к РФ = лучший пинг, набирает больше)
- FAL: ~25 юзеров
- AMS: ~25 юзеров
- FRA: ~25 юзеров
- Каждый сервер загружен на ~25% → огромный запас

#### Один сервер заблокирован
- HEL отвалился → API убирает из subscription за 30 сек
- Happ на клиентах автоматически переключается на следующий
- 100 юзеров перераспределяются: FAL ~35, AMS ~35, FRA ~30
- Каждый сервер загружен на ~33% → всё ещё комфортно
- Бот алертит админу: «HEL недоступен, осталось 3 сервера. Купи новый!»

#### Два сервера заблокированы
- Осталось AMS + FRA
- 100 юзеров → 50/50
- Каждый на ~50% нагрузки → работает, но без запаса
- Бот: «КРИТИЧНО: осталось 2 сервера! Новый сервер нужен СЕЙЧАС!»
- Скрипт деплоя нового сервера: 5 минут

#### Белые списки (шатдаун мобильного)
- API добавляет relay-серверы в subscription
- Юзеры на мобильном: Happ автоматически выбирает relay (доступен)
- Юзеры на проводном: остаются на зарубежных (быстрее)

### Алерты для админа (Telegram)
- «Сервер HEL: CPU 80%+» → предупреждение о перегрузке
- «Сервер HEL: недоступен 2 минуты» → вероятная блокировка
- «Осталось 3/4 серверов» → нужен новый
- «Осталось 2/4 серверов» → КРИТИЧНО
- «Трафик HEL: осталось 15%» → скоро лимит
- Еженедельный отчёт: нагрузка, юзеры, доходы, статус серверов

## 5. Протоколы и конфигурация

### Основной: VLESS + Reality + TCP
```
Протокол: VLESS
Шифрование: none (шифрует Reality)
Flow: xtls-rprx-vision
Security: reality
Fingerprint: chrome (uTLS)
SNI: малоизвестный домен (НЕ Google/Yandex/VK)
Порт: 443
```

### Fallback: VLESS + Reality + XHTTP
Если TCP начнёт палиться ТСПУ — переключаем на XHTTP.
XHTTP маскирует трафик под обычные HTTP/2 запросы с padding.
Конфигурация аналогична, меняется только network: "xhttp".

### Для relay (уровень 2): VLESS chain
```
Клиент → Relay (VLESS Reality, SNI из белого списка)
Relay → Exit (VLESS Reality, малоизвестный SNI)
```

### MTProto Proxy (Telegram)
На каждом сервере: mtg (Docker, 5MB)
Порт: 8443
Маскировка: FakeTLS под www.google.com (для MTProto это ОК,
т.к. MTProto — отдельный протокол, не проходит через ТСПУ как VPN)

## 6. Клиентская часть

### MVP: Happ (готовое приложение)
- App Store: Happ - Proxy Utility (iOS 15+)
- Google Play: Happ - Proxy Utility (Android 5+)
- macOS: десктопный клиент через GitHub

Happ поддерживает:
- VLESS Reality, VMess, Trojan, Shadowsocks, Hysteria2
- Подписки (subscription URL) с автообновлением
- Автоподключение к самому быстрому серверу
- Fallback URL если основной домен заблокирован
- Скрытые и зашифрованные подписки

### Как юзер подключается
1. Скачивает Happ из App Store / Google Play
2. Получает от бота subscription URL
3. В Happ: + → Подписка → вставляет URL
4. Happ сам скачивает список серверов, выбирает лучший, подключает
5. При блокировке → Happ сам обновляет список и переключает

### Фаза 2: Форк AmneziaWG / Happ
После MVP — свой клиент с брендингом:
- Автообновление конфигов через API (без ручного импорта)
- Healthcheck: пинг каждые 30 сек, автопереключение за 2 сек
- Встроенная оплата
- Push-уведомления о статусе

## 8. Маскировка и безопасность

### Зарубежные серверы
- Reality masquerade: при зондировании отдаёт реальный сайт-донор
- Уникальные ключи X25519 на каждом сервере
- SNI подбирается через RealiTLScanner для конкретной подсети
- Каждый юзер — уникальный UUID
- Нет логов трафика

### Российский relay
- Реальный работающий сайт (IP-чекер / speedtest)
- Nginx фронтенд: легитимные запросы → сайт, VPN-трафик → XRay
- Минимальный трафик через relay (только мобильный шатдаун)
- Два relay у разных хостеров для резерва
- Аккаунты зарегистрированы на разные данные

### Subscription URL
- Раздаётся через Telegram-бот (не через сайт, который могут заблокировать)
- Зашифрован (Happ поддерживает encrypted subscriptions)
- Fallback URL на случай блокировки основного домена
- Токен уникален для каждого юзера
