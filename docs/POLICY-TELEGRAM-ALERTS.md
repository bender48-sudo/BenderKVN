# Политика: что допускается в Telegram-алертах (P2-MON-03)

Цель: админ получает **достаточно контекста для диагностики**, без **лишних метаданных** о пользователях и без **секретов**.

## Разрешено в тексте алерта

- Имя проверки (`xray_lv_443`, `subscription`, `panel`, …).
- HTTP-коды, **обезличенные** URL **публичных** эндпоинтов (панель, подписка — уже в проде известны).
- Указание хоста/роли (**LV**, **AMS**, **NL**) и **не секретных** IP из §1 бэклога / runbook.
- Короткая **команда для разбору** без вставки токенов (`docker logs …`, `systemctl status …`).

## Запрещено или недопустимо по умолчанию

- **Bearer / JWT / BOT_TOKEN / пароли** — никогда; даже «первые 10 символов».
- **Email пользователей**, **UUID подписки** клиента, **ключи** подписки (**shortId**), адреса клиентов.
- Полный дамп ответа API; при необходимости — только **код ответа** и **обрезанное** поле из JSON (без PII).

## Мониторы в репозитории

Скрипты **`monitor.sh`**, **`daily-report.sh`**, **`balancer.sh`**, **`ru-monitor.py`**, **`selfsteal-monitor.py`** — при добавлении новых алертов сверяйтесь с этим файлом. При сомнении — **меньше данных в TG**, больше в **приватный лог** на сервере (`/var/log/bvpn-*.log`).

## Связь с P3

Пользовательские тексты и онбординг (**P3-UX-***) регулируются отдельно; это документ **про админские уведомления**.
