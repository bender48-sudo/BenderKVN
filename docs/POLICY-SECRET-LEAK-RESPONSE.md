# Политика: утечка секрета в транскрипт, скриншот или лог (P2-SEC-LOG-01)

Если **BOT_TOKEN**, **JWT панели**, **REMNA API token**, пароль БД или иной **живой секрет** мог оказаться в:

- транскрипте чата (Cursor, иной ИИ),
- общем Slack/Discord/Telegram,
- скриншоте IDE,
- **публичном** gist / pastebin / тикете без редактирования,

— **до проверки** считаем ситуацию **компрометацией** и действуем по **минимуму** ниже.

## Минимальный порядок

1. **Прекратить** дальнейшее распространение (удалить сообщение/файл там, где возможно).
2. **Не** повторять секрет в новых сообщениях при эскалации.
3. **Ротация** затронутого секрета по соответствующему runbook (**`docs/SECRETS.md`**, **`docs/RUNBOOK-REMNA-API-TOKEN.md`**, ротация бота в панели провайдера и т.д.).
4. **Redeploy / перезапуск** всех потребителей (контейнеры с `.env`, cron с `balancer.env`, …).
5. Точечный **grep** по типичным путям (логи сборок CI, копии репо), что строка не утекла ещё куда-то.

## Типичные артефакты для grep

- `eyJ` (начало JWT),
- паттерн Telegram bot `^[0-9]+:[A-Za-z0-9_-]{20,}$` в нетипичных файлах,
- фрагменты `REMNA_API_TOKEN=`, `BOT_TOKEN=` вне **`.secrets/`**.

## Связь с коммерческим бэклогом

Критерий **P2-SEC-LOG-01** закрыт **политикой в репо** + соблюдение при инцидентах; отдельный «юридический» текст — при необходимости вне git.
