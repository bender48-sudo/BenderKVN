# Политика: одна задача — один коммит — новая сессия

Согласовано для фазы роста (**10k** к лету 2026): владелец возвращается к агенту **между** задачами; агент **не** съедает очередь целиком за один чат.

## Источники

| Документ | Роль |
|----------|------|
| **`docs/BACKLOG-QUEUE.md`** | Линейная очередь **Q001…**; ровно один **`NEXT`** |
| **`docs/COMMERCIAL-BACKLOG.md`** | Определения **Done when**, §5.1 Red team, §10 пороги |
| **`docs/POLICY-BACKLOG-ORDER.md`** | Продукт раньше UX; коммерция до GTM |

## Цикл (обязательный)

1. Открыть **`BACKLOG-QUEUE.md`** → взять **`NEXT`**.
2. Сделать только эту задачу (runbook из колонки таблицы).
3. Проверить (**Verify**).
4. Обновить очередь (**DONE** / новый **NEXT**) + при необходимости **§12** бэклога.
5. **`git commit`** (и **`push`**, если принято в репо).
6. **Стоп.** Следующий **Q** — только после возврата владельца (или явной фразы «продолжай очередь»).

## Для агента Cursor

Правило **`.cursor/rules/sequential-backlog.mdc`** (`alwaysApply: true`).

## Для владельца

После push проверьте прод/smoke по **Verify** строки. В новом чате напишите, например: «выполни **NEXT** из **BACKLOG-QUEUE**» — агент не должен перескакивать на Q003, пока Q002 не **DONE**.

## Накат AMS

Любой деплой compose/env на AMS — gate **`RUNBOOK-AMS-SAFE-DEPLOY`** (не отдельный Q, но обязателен перед шагами, трогающими AMS).
