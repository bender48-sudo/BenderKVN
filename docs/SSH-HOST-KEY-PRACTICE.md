# SSH: known_hosts и `StrictHostKeyChecking` (P2-SSH-01)

Цель: **предсказуемое** поведение автоматизации (cron, `drift-check`, деплой) и **снижение** риска MITM без «вечного `-o StrictHostKeyChecking=no` на всё».

## Хосты продакшена

| Хост / алиас в `~/.ssh/config` | Назначение | Рекомендация ключа хоста |
|--------------------------------|------------|----------------------------|
| **bvpn-lv** (Latvia) | Мониторы, скрипты `/opt/scripts/` | Один раз принять ключ (**`accept-new`** или ручной `ssh-keyscan` → `known_hosts`), далее **pin** |
| **bvpn-ams** (Amsterdam) | Панель, sub, бот | То же |
| **bvpn-nl** (NL) | Нода, watchdog | То же |

Точные IP/порты — в **`docs/SECRETS.md`** / вашем частном контуре, не дублировать в публичный репо без необходимости.

## Практика в репозитории

- **Интерактивный** первый заход: `StrictHostKeyChecking=accept-new` (как в **`ops/deploy-*-lv.ps1`**) — ключ записывается один раз.
- **Cron и скрипты на сервере** (LV→AMS): предпочтительно **`UserKnownHostsFile`** с файлом, куда занесён **один** ожидаемый ключ, либо системный **`/etc/ssh/ssh_known_hosts`** под контролем конфиг-менеджмента.
- Избегать **`StrictHostKeyChecking=no`** в проде для **рутиновых** SSH; если осталось в legacy-скрипте — завести задачу на замену после pin ключа.

## Оператору

При смене VPS/IP — обновить **known_hosts** на машинах, с которых идёт SSH (LV, NL, рабочая станция с `drift-check`), иначе проверки упадут с **REMOTE HOST IDENTIFICATION HAS CHANGED** — это **нормальная** защита, не «глушить» через `no`.
