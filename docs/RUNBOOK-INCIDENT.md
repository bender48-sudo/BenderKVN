# Runbook инцидента (внутренний, 1 стр.)

**P1-PRO-02**. Не содержит секретов и токенов — только имена ресурсов и порядок действий.

## Роли и каналы

**Имена и контакты дежурных** (TG, телефон) — **не** в этом репозитории: ведите в корпоративной базе (**Notion**/private wiki): кто первый контур поддержки и кто эскалирует в инфру.

| Роль | Назначение |
|------|------------|
| **Первый ответ поддержки / дежурный** | Принять сигнал, подтвердить масштаб, шаблон ответа пользователю. |
| **Оператор инфры** | SSH на хосты, `monitor`/`drift`/compose по **`docs/DEPLOY.md`**. |

**Целевое время первого ответа пользователю** — зафиксировать локально командой (рекомендация: до **60 минут** в рабочее окно; вне её — сообщение автоматически/заготовка «разбираемся»).

## Быстрая классификация

1. **«Не коннектится VPN»** — запросить у пользователя: клиент, версия, скрин ошибки; предложить **обновить подписку** и переподключиться (см. **`docs/FAQ.md`**).
2. **«Не открывается сайт / только RU»** — проверить, не ожидает ли пользователь **direct** для RU‑ресурса; сверка с **`docs/RU-BYPASS.md`**.
3. **Массовый даун** — панель, подписка, ноды (см. проверки ниже).

## Проверки без логина в панель (с VPN с админской машины)

- Подписка (как в `monitor.sh`): HTTP **200** на smoke URL подписки (хост **`SUB_PUBLIC_ORIGIN`** в **`ops/site.env.example`**).
- Панель: главная по **`PANEL_URL`** с валидным TLS (или с рабочей станции через `curl -fsS`).

Детальнее: **`ops/p0-audit.sh`** на LV/AMS (если синхронизирован с продом).

## Проверки с API-панели (нужен `PANEL_TOKEN`)

- **`python ops/panel_api.py nodes`** — ноды `connected=True`.
- **`python ops/probe_routing.py`** — для выборки пользователей: **`degenerate rules = 0`**.

Шаблон подписки ломался после PATCH — см. rollback в **`docs/HAPP-MATRIX.md`**.

## Логин в панель Remnawave (саппорт / операторы)

Клиент браузера иногда показывает **Forbidden** и код вроде **E000** не только из‑за прокси или «блокировки», но и при **неверном пароле** (**HTTP 403**, не **401**). Перед диагностикой сети и IP:

1. Попросите оператора **повторить пароль вручную** (буфер обмена, раскладка, лишний пробел).  
2. Если пароль восстановили через **Rescue CLI** официальной документации панели — убедитесь, что входят в **правильный** URL панели (`PANEL_DOMAIN` / `FRONT_END_DOMAIN`, без чернового зеркала).  
3. Только если пароль точно верный — смотреть ACL, географию, CDN и лог панели.

Задача в бэклоге: **`P3-OPS-SUPPORT-REMNA-LOGIN`** (эта секция её закрывает по документации).
